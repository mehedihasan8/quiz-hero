Difference cookies, local storage and session storage

Cookies, Session Storage, and Local Storage are all ways to store data in a user's browser, but they have different characteristics and use cases:

1. Cookies: Cookies are small text files that are stored on the client-side by the browser. They are commonly used to store user preferences, session IDs, and other small amounts of data.
2. Session Storage: Session Storage is a mechanism for storing key-value pairs in the browser, similar to local storage. However, Session Storage is only available for the duration of a user's session on a website. Once the user closes the browser, the data stored in Session Storage is deleted.
3. Local Storage: Local Storage is another mechanism for storing key-value pairs in the browser, Some like that Session Storage, but Local Storage persists even after the user closes the browser. Local Storage is often used to store user preferences, caching data, or other data that needs to be accessed across multiple sessions. However, Local Storage has a maximum size limit that varies depending on the browser.

How does the Event Loop work?

The event loop is a mechanism that continuously checks the JavaScript execution stack and the task queue to determine what code needs to be executed next. Here's how it works

1. JavaScript has a single thread of execution, which means that it can only do one thing at a time. When a function is called, it gets added to the call stack, which is a data structure that keeps track of where the program is in its execution.
2. When an asynchronous operation, such as a setTimeout or an HTTP request, is called, it gets added to the task queue, which is a data structure that holds tasks that are waiting to be executed.
3.Once the call stack is empty, the event loop checks the task queue to see if there are any tasks waiting to be executed. If there is a task in the queue, the event loop moves it from the task queue to the call stack, and the function associated with the task is executed.
4. As the function is executing, any synchronous code in the function gets added to the call stack. When the function finishes executing, the call stack is empty again, and the event loop checks the task queue for the next task.
5. This process continues until there are no more tasks in the task queue, at which point the event loop stops checking for tasks and the program ends.
